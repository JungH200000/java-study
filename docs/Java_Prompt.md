## [요청 사항]

### 1) 기술적 정확성 & 명료성

- `[원본 글]`에 등장하는 Java 개념과 코드를 **면밀히 검토**해라. (예: **JVM/JRE/JDK, 기본 문법, 자료형/연산, 제어문, 메서드, OOP(클래스/객체/상속/다형성/캡슐화), 인터페이스/추상클래스, 패키지/모듈(JPMS), 예외 처리, 제네릭, 컬렉션 프레임워크, I/O & NIO.2, 람다/스트림, 동시성(스레드/Executor/synchronized/volatile), 메모리 모델 개요, JDBC/ORM(필요 시), 빌드 도구(Maven/Gradle)(필요 시), 테스트(JUnit)(필요 시)**)
- **오개념/부정확한 표현**을 바로잡되, **새로운 내용의 임의 추가는 금지**한다.
- **핵심 용어를 먼저 정확히 정의**하고, **짧고 선명한 한 문장 요약**을 곁들여라.

### 2) 논리적 흐름 & 구조

- 문서가 가능한 한 **입문 학습 흐름**을 따르도록 정리/보완해라. (필요할 때만 재배치)

    - **기초 문법 → 자료형/연산 → 제어문/메서드 → OOP(클래스/상속/다형성) → 인터페이스/추상클래스 → 패키지/모듈 → 예외 처리 → 제네릭/컬렉션 → I/O/NIO.2 → 람다 & 스트림 → 동시성 → (선택) JDBC/ORM → (선택) 빌드/테스트**

- 문단 간 연결을 **자연스럽게** 하고, 중복은 통합하되 **원문 정보는 보존**해라.

### 3) 입문자 친화 설명(최우선)

- **정의 우선 원칙:** 새 용어/개념은 **간단한 정의**를 맨 앞에 둔다.
- **친절한 부연:** 정의 뒤에 **직관적 비유/짧은 예시**로 이해를 돕는다.
- **코드 설명 강화:**

    - 각 코드 블록마다 **핵심 포인트에 주석**과 **라인별/블록별 설명**을 붙인다.  
      (예: 메서드 시그니처 의미, 접근 제어자 의도, 예외 발생/전파, 동시성에서의 공유 상태, 제네릭 타입 파라미터 역할 등)
    - 초보자에게 난해한 부분(예: **오버로딩 vs 오버라이딩, 참조/값 의미, equals/hashCode 규약, 제네릭 타입 소거, synchronized/volatile 차이, 스트림 중간/종단 연산**)은 **짧은 보충 설명**을 덧붙인다.

- **불필요한 난해 표현은 지양**하고, **간단한 문장**을 사용한다.

### 4) Markdown 형식 최적화

- **제목/부제목**(H1~H3), **코드 블록(```java)**, **목록, 표, 인용** 등을 활용해 **가독성**을 높인다.
- 긴 코드/긴 문단은 **적당히 분할**하고, **중요 요점은 목록/강조**로 정리한다.
- **실행 환경/버전**(예: _JDK 17, Maven/Gradle 버전, IDE_)이 섞일 경우 **섹션을 분리**하고, 필요한 범위에서만 명시한다.

### 5) 동작 순서·흐름/구조 설명(필요 시)

- `[원본 글]`에 **JVM 동작, 예외 전파, 스레드 실행 흐름, 스트림 파이프라인 처리 순서** 등의 **절차/흐름**이 있으면,  
  각 단계 **키워드 설명**을 짧게 붙이고 **문맥에서의 역할**을 1~2문장으로 덧붙인다.

### 6) 핵심 요약(필수)

- 문서 맨 끝에 **‘핵심 요약’** 섹션을 추가한다.
- `[원본 글]` 기반으로 **3~5줄** 내에서 **핵심 개념과 흐름**을 간결히 정리한다.
